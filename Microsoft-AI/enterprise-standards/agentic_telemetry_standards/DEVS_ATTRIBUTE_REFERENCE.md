# Tracing Attribute Reference: Standard vs. Developer-Defined

**Audience:** All engineers building on Azure AI Foundry  
**Purpose:** Clearly identify which attribute names, span names, and metrics are fixed by an open standard or SDK ‚Äî and which you write yourself.

---

## The Core Distinction

Every piece of data attached to a trace falls into one of three buckets:

| Bucket | Who defines the name | Can you rename it? | Where it comes from |
|---|---|---|---|
| **[1] OTel / GenAI Semantic Convention**  | OpenTelemetry specification | ‚ùå No ‚Äî renaming breaks tooling | SDK auto-instrumentation |
| **[2] Org-Standard Contract** | This repo / your org | ‚öÝÔ∏è No ‚Äî renaming breaks shared dashboards | `GlobalEnrichmentProcessor` or per-span `set_attribute()` calls |
| **[3] Developer-Defined** | You | ‚úÖ Yes ‚Äî fully your choice | Your own `set_attribute()` calls |

The rest of this document maps every attribute, span name, and metric to one of these three buckets.

## 1. Span Names

Span names are the labels that appear in the trace waterfall in Application Insights and the Foundry Portal. Some are emitted by the SDK automatically; others you write in your code.

### SDK-emitted span names (`azure-ai-agents`) ‚Äî CANNOT be renamed

These are generated by `AIAgentsInstrumentor` and follow the pattern `<resource>.<operation>`. The names are hardcoded in the SDK.

| Span name | Triggered by |
|---|---|
| `agents.create` | `client.create_agent(...)` |
| `agents.get` | `client.get_agent(agent_id)` |
| `threads.create` | `client.threads.create()` |
| `messages.create` | `client.messages.create(...)` |
| `messages.list` | `client.messages.list(...)` |
| `runs.create` | First step inside `create_and_process()` |
| `start_thread_run` | `create_and_process()` ‚Äî run initiation |
| `get_thread_run` | `create_and_process()` ‚Äî each polling tick |
| `process_thread_run` | `create_and_process()` ‚Äî final processing |

> **These names are your filter targets.** The `IMPORTANT_OPERATIONS` and `SUPPRESS_OPERATIONS` lists in `ConsoleSpanExporter` use substrings of these exact names.

### SDK-emitted span names (Microsoft Agent Framework) ‚Äî CANNOT be renamed

| Span name | Triggered by |
|---|---|
| `invoke_agent {agent_name}` | Any `agent.run(...)` call |
| `chat {model_name}` | Each LLM call inside an agent run |
| `execute_tool {function_name}` | Each tool/function invocation |

### Org-standard span names ‚Äî FIXED BY CONVENTION (do not deviate)

These are spans you write, but the names are fixed by the org standard. Deviating breaks the shared KQL dashboards.

| Span name | Pattern | Example | SDK |
|---|---|---|---|
| Session root | `session` | `session` | Both |
| Turn | `turn-{integer}` | `turn-1`, `turn-2` | Both |
| Agent lifecycle | `agent.{action}` | `agent.get_or_create` | `azure-ai-agents` |
| MAF scenario root | `Scenario: {description}` | `Scenario: claims-triage` | MAF |

### Developer-named spans ‚Äî YOUR CHOICE

Spans you write for sub-operations within a turn. Follow the `snake_case` verb phrase convention, but the names are yours.

```python
# Examples from the reference implementation ‚Äî you can name these anything
with tracer.start_as_current_span("add_user_message"):   ...
with tracer.start_as_current_span("agent_run"):          ...
with tracer.start_as_current_span("get_response"):       ...
with tracer.start_as_current_span("validate_claim"):     ...   # your domain logic
with tracer.start_as_current_span("enrich_user_context"): ...  # your domain logic
```

## 2. Resource Attributes (process-level)

Resource attributes describe the *process*, not a single span. They are set once at startup and attached to every span automatically. Their names come from the OpenTelemetry Resource Semantic Conventions ‚Äî **you cannot rename them**, but **you supply the values** via environment variables.

| Attribute name | Fixed? | Source | Set via |
|---|---|---|---|
| `service.name` | ‚úÖ Fixed (OTel spec) | Your value | `OTEL_SERVICE_NAME` env var |
| `service.version` | ‚úÖ Fixed (OTel spec) | Your value | `OTEL_SERVICE_VERSION` env var |
| `service.instance.id` | ‚úÖ Fixed (OTel spec) | Auto-generated | SDK / Azure Monitor |
| `telemetry.sdk.name` | ‚úÖ Fixed (OTel spec) | `"opentelemetry"` | SDK (automatic) |
| `telemetry.sdk.version` | ‚úÖ Fixed (OTel spec) | Package version | SDK (automatic) |
| `telemetry.sdk.language` | ‚úÖ Fixed (OTel spec) | `"python"` | SDK (automatic) |
| `cloud.role` | ‚úÖ Fixed (Azure spec) | `service.name` value | Azure Monitor exporter |
| `cloud.roleInstance` | ‚úÖ Fixed (Azure spec) | Hostname | Azure Monitor exporter |

**How they map to Application Insights:**

In Application Insights, `service.name` becomes the **Cloud role name** visible in the Application Map. `service.instance.id` becomes **Cloud role instance**. These are the primary grouping dimensions in the portal.

## 3. GenAI Semantic Convention Attributes ‚Äî Bucket 1

These attribute names are defined by the [OpenTelemetry Semantic Conventions for Generative AI Systems](https://opentelemetry.io/docs/specs/semconv/gen-ai/). They are emitted automatically by `AIAgentsInstrumentor` (azure-ai-agents) and MAF's built-in instrumentation.

**You must not set these manually** ‚Äî the SDK already sets them. Duplicating them creates conflicting values in Application Insights.

### Always present (no env var required)

| Attribute | Type | Example value | Which SDK emits it |
|---|---|---|---|
| `gen_ai.system` | string | `"az.ai.agents"` | azure-ai-agents |
| `gen_ai.operation.name` | string | `"invoke_agent"`, `"chat"`, `"execute_tool"` | Both |
| `gen_ai.request.model` | string | `"gpt-4o"`, `"Llama-4-Maverick..."` | Both |
| `gen_ai.response.model` | string | `"gpt-4o-2024-11-20"` | azure-ai-agents |
| `gen_ai.usage.input_tokens` | int | `842` | Both |
| `gen_ai.usage.output_tokens` | int | `317` | Both |
| `gen_ai.agent.name` | string | `"ClaimsAgent"` | MAF only |
| `gen_ai.agent.id` | string | `"claims-agent"` | MAF only (from `Agent(id=...)`) |

> **MAF note on `gen_ai.agent.id`:** This is a GenAI semantic convention attribute, but its *value* is determined by what you pass to `Agent(id="...")`. The **name** `gen_ai.agent.id` is fixed; the **value** is yours. Use a stable, lowercase-kebab-case string that matches the agent registered in the Foundry Portal.

### Present only when content recording is enabled

These attributes appear on spans only when the corresponding environment variable is set to `true`. See [Section 6](#6-conditional-attributes-content-recording) for the PII policy.

| Attribute | Content it holds | Controlled by |
|---|---|---|
| `gen_ai.prompt` | Full user message text sent to the model | `AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true` |
| `gen_ai.completion` | Full assistant response received | Same |
| `gen_ai.tool.arguments` | Arguments passed to a tool/function | `ENABLE_SENSITIVE_DATA=true` (MAF) |
| `gen_ai.tool.result` | Return value from a tool/function | `ENABLE_SENSITIVE_DATA=true` (MAF) |

## 4. Org-Standard Contract Attributes ‚Äî Bucket 2

These attribute names are **defined by this org's tracing standard**, not by OpenTelemetry. They are fixed in the sense that renaming them would break the shared KQL dashboards and Azure Workbooks. Every team **must** use these exact names.

Most of these are injected automatically on every span by `GlobalEnrichmentProcessor` from environment variables. You do not need to call `set_attribute()` for them manually ‚Äî the processor does it for you.

### Injected automatically by `GlobalEnrichmentProcessor` (set env vars, not code)

| Attribute | Type | Source env var | Example |
|---|---|---|---|
| `app.service.name` | string | `SERVICE_NAME` | `"claims-processor"` |
| `app.version` | string | `APP_VERSION` | `"1.4.2"` |
| `deployment.env` | string | `ENVIRONMENT` | `"prod"`, `"staging"`, `"dev"` |
| `org.team` | string | `TEAM_NAME` | `"phoenix"` |
| `org.cost_center` | string | `COST_CENTER` | `"cc-1234"` |

These five attributes appear on **every single span** your process emits ‚Äî no per-span code required.

### Must be set manually with `set_attribute()` ‚Äî required by org contract

These are not injectable from env vars because their values are runtime-dynamic (e.g., they change per user session or per turn). You must call `span.set_attribute(name, value)` explicitly in your code.

#### On the session root span

| Attribute | Type | Example | Notes |
|---|---|---|---|
| `session.id` | string | UUID v4 | Generate with `str(uuid.uuid4())` |
| `agent.id` | string | `"asst_abc123"` | The Foundry agent ID |
| `thread.id` | string | `"thread_xyz789"` | The Foundry thread ID |
| `total.turns` | int | `3` | Set at session end |
| `user.id` | string | hashed value | **Never** a raw email in staging/prod |

#### On each turn span (`turn-{N}`)

| Attribute | Type | Example | Notes |
|---|---|---|---|
| `turn.number` | int | `1` | Sequential from 1 |
| `turn.latency_ms` | float | `1823.4` | End-to-end turn duration |
| `run.status` | string | `"completed"` | Final run state |
| `response.length` | int | `342` | Character count of assistant reply |
| `user.query` | string | `"What is my deductible?"` | **Disable in prod if PII** |

#### On the agent run span

| Attribute | Type | Example | Notes |
|---|---|---|---|
| `run.id` | string | `"run_def456"` | Foundry run ID ‚Äî use in Transaction Search |

## 5. Developer-Defined Attributes ‚Äî Bucket 3

These are attributes you invent for your own domain logic. Their names are entirely your choice ‚Äî they are not validated by any spec or shared dashboard.

### From the reference implementation

These appear in `foundry_agent_tracing_patterns.py` as examples of developer-defined attributes:

| Attribute | Set on span | What it captures |
|---|---|---|
| `demo.type` | session | Identifies the demo scenario (`"tracing_patterns"`) |
| `agent.reused` | `agent.get_or_create` | `True` if cached agent was reused, `False` if new |

> `agent.reused` is a good example of a developer-defined attribute that becomes operationally useful: you can filter Application Insights on `customDimensions["agent.reused"] == false` to measure cold-start frequency.

### From the claims-triage reference app

| Attribute | Set on span | What it captures |
|---|---|---|
| `intent` | `orchestrate` | Classified intent from the orchestrator |
| `worker.agent` | `delegate.<intent>` | Which worker agent handled the turn |
| `claim.id` | domain spans | Business identifier for the claim being processed |

### Org-wide optional attributes (use when applicable)

These are not required by the contract but are recognized by the shared dashboards if present:

| Attribute | Type | Example | Purpose |
|---|---|---|---|
| `experiment.variant` | string | `"chain-of-thought-v2"` | A/B testing ‚Äî which prompt variant ran |
| `org.product` | string | `"claims-ai"` | Product line (more granular than team) |
| `org.squad` | string | `"phoenix"` | Squad within a team |
| `feature.flag` | string | `"multi-agent-routing"` | Feature flag that was active during this session |

### Naming rules for developer-defined attributes

1. Use `namespace.attribute_name` dot-notation (e.g., `claim.id`, not `claimId`)
2. Use lowercase and underscores within the local part (e.g., `claim.policy_number`)
3. Do **not** use `gen_ai.*` ‚Äî that prefix is reserved for the OTel semantic convention
4. Do **not** use `service.*`, `telemetry.*`, or `cloud.*` ‚Äî reserved for OTel resource semantics
5. Do **not** put dynamic user data (query text, response text) into attribute names ‚Äî only into values, and only when content recording is appropriate

## 6. Conditional Attributes: Content Recording

Content recording attributes are Bucket 1 (GenAI semantic convention names, fixed) but are **gated behind environment variables**. Their presence or absence is controlled entirely by configuration, not code.

| Attribute name | Fixed? | Present when | SDK |
|---|---|---|---|
| `gen_ai.prompt` | ‚úÖ Fixed | `AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true` | azure-ai-agents |
| `gen_ai.completion` | ‚úÖ Fixed | `AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true` | azure-ai-agents |
| `gen_ai.prompt` | ‚úÖ Fixed | `ENABLE_SENSITIVE_DATA=true` | MAF |
| `gen_ai.completion` | ‚úÖ Fixed | `ENABLE_SENSITIVE_DATA=true` | MAF |
| `gen_ai.tool.arguments` | ‚úÖ Fixed | `ENABLE_SENSITIVE_DATA=true` | MAF |
| `gen_ai.tool.result` | ‚úÖ Fixed | `ENABLE_SENSITIVE_DATA=true` | MAF |

### Environment policy matrix

| Environment | `AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED` | `ENABLE_SENSITIVE_DATA` | Effect |
|---|---|---|---|
| dev | `true` | `true` | Full prompt + response text on every span |
| staging | `false` (default) | `false` (default) | No text content ‚Äî token counts only |
| prod (internal) | `false` (default) | `false` (default) | No text content ‚Äî token counts only |
| prod (external / regulated) | Must not be set | Must not be set | No content, enforced at CI/CD level |

> When content recording is off, the `gen_ai.usage.input_tokens` and `gen_ai.usage.output_tokens` attributes are **still present** ‚Äî you always have token counts for cost attribution, regardless of PII policy.

## 7. Metrics: Standard vs. Custom

Unlike spans (which record a single operation), metrics are continuous numeric measurements. The SDKs emit some automatically; you can add your own via the OpenTelemetry Metrics API.

### MAF built-in metrics ‚Äî CANNOT be renamed (Bucket 1)

These are emitted by the `agent_framework` package automatically when `ENABLE_INSTRUMENTATION=true`. They appear in Application Insights `customMetrics` table.

| Metric name | Type | Unit | What it measures |
|---|---|---|---|
| `gen_ai.client.operation.duration` | Histogram | seconds | End-to-end LLM call duration per `chat` span |
| `gen_ai.client.token.usage` | Histogram | tokens | Tokens per LLM call (with `gen_ai.token.type` dimension: `input` / `output`) |
| `agent_framework.function.invocation.duration` | Histogram | seconds | Tool/function execution duration per `execute_tool` span |

> `azure-ai-agents` does **not** emit built-in OTel metrics. Duration and token data for that SDK come from span attributes (`gen_ai.usage.input_tokens`, span `end_time - start_time`), not from the `customMetrics` table.

### Developer-defined metrics (optional, Bucket 3)

You can emit your own metrics using the OTel Metrics API. Names are your choice ‚Äî follow the `namespace.metric_name` convention.

```python
from opentelemetry import metrics

meter = metrics.get_meter(__name__)

# Counter ‚Äî counts discrete events
claim_counter = meter.create_counter(
    "claims.processed",
    unit="claims",
    description="Number of claims processed by the agent",
)
claim_counter.add(1, {"claims.type": "auto", "deployment.env": os.getenv("ENVIRONMENT")})

# Histogram ‚Äî distribution of values (e.g., response lengths)
response_length_hist = meter.create_histogram(
    "agent.response.length",
    unit="characters",
    description="Character length of agent responses",
)
response_length_hist.record(len(response), {"agent.id": agent_id})
```

These appear in Application Insights `customMetrics` and are queryable via KQL.

## 8. Span Status & Error Fields

Span status is a first-class OTel concept with a fixed structure ‚Äî it is not a custom attribute.

### Fixed fields (cannot be renamed)

| Field | Type | Values | Set via |
|---|---|---|---|
| `status.code` | enum | `UNSET`, `OK`, `ERROR` | `span.set_status(trace.Status(trace.StatusCode.ERROR, description))` |
| `status.description` | string | Free text | Second argument to `trace.Status(...)` |

In Application Insights:
- Spans with `StatusCode.ERROR` appear as `success = false` in the `dependencies` table
- The `status.description` appears as the `resultCode` field

### How to set error status correctly

```python
from opentelemetry import trace

with tracer.start_as_current_span("agent_run") as run_span:
    with tracer.start_as_current_span("turn-1") as turn_span:
        try:
            run = client.runs.create_and_process(...)
            if run.status == "failed":
                error_msg = f"Run failed: {run.last_error}"
                # Mark BOTH the inner span and the parent turn span
                run_span.set_status(trace.Status(trace.StatusCode.ERROR, error_msg))
                turn_span.set_status(trace.Status(trace.StatusCode.ERROR, error_msg))
        except Exception as e:
            run_span.record_exception(e)   # attaches stack trace to the span
            run_span.set_status(trace.Status(trace.StatusCode.ERROR, str(e)))
            raise
```

`record_exception(e)` is a fixed OTel API call that attaches the exception type, message, and stack trace as a span event. The event name `exception` and its attributes (`exception.type`, `exception.message`, `exception.stacktrace`) are all defined by the OTel spec ‚Äî you cannot rename them.

## Summary Decision Tree

When you are about to call `span.set_attribute(name, value)`, ask:

```
Is this attribute name defined in the OTel GenAI semantic conventions?
   (i.e., does it start with gen_ai.*, service.*, cloud.*, exception.*)
        ‚îÇ
        ‚îú‚îÄ YES ‚Üí Do NOT set it manually. The SDK already sets it.
        ‚îÇ         You would create a duplicate or conflicting value.
        ‚îÇ
        ‚îî‚îÄ NO
              ‚îÇ
              Does this attribute appear in the org contract table above?
              (app.service.name, deployment.env, org.team, turn.number, etc.)
                    ‚îÇ
                    ‚îú‚îÄ YES, process-level ‚Üí Set it as an env var.
                    ‚îÇ   GlobalEnrichmentProcessor injects it on every span.
                    ‚îÇ   You do not need to call set_attribute() at all.
                    ‚îÇ
                    ‚îú‚îÄ YES, runtime-dynamic ‚Üí Call set_attribute() exactly once
                    ‚îÇ   on the correct span. Use the exact org-contract name.
                    ‚îÇ
                    ‚îî‚îÄ NO ‚Üí This is a developer-defined attribute. 
                            Name it using namespace.attribute_name convention.
                            Avoid gen_ai.*, service.*, cloud.*, exception.* prefixes.
```
